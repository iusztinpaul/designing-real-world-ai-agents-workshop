# LinkedIn Writer — Evaluation Harness

This harness evaluates the quality of LinkedIn posts generated by the LinkedIn Writer MCP server. It uploads labeled dataset splits to Opik, runs a `BinaryLLMJudgeMetric` (backed by Gemini) against each post, and computes F1 between judge verdicts and human-assigned labels for offline splits. For online splits it generates posts on the fly via the writing workflow and judges them immediately, emulating a production pipeline. The harness covers three workflows: offline dev alignment, offline held-out test validation, and online generation + judgment.

See also: [LinkedIn Writer MCP Server](../README.md) | [Deep Research MCP Server](../../research/README.md)

## Prerequisites

- Python 3.12 or later (see [`.python-version`](../../../.python-version))
- `uv` for virtual environment and dependency management ([installation guide](https://docs.astral.sh/uv/getting-started/installation/))
- GNU Make (pre-installed on macOS/Linux; `choco install make` on Windows)
- `OPIK_API_KEY` — **required** for all eval workflows (unlike the rest of the system, where Opik is optional). Evals write dataset items and evaluation results directly to Opik.
- Google AI Studio API key ([aistudio.google.com/apikey](https://aistudio.google.com/apikey)) — required because the LLM judge calls Gemini.
- The `datasets/linkedin_paul_iusztin/` directory must be accessible from `implement_yourself/`. The repository ships a symlink at `implement_yourself/datasets → ../datasets`. For a fresh standalone clone of `implement_yourself/`, create it manually:

  ```bash
  # Run from inside implement_yourself/
  ln -s ../datasets ./datasets
  ```

## Installation

From the `implement_yourself/` project root, run:

```bash
uv sync
```

This resolves all dependencies declared in `pyproject.toml` and creates an isolated `.venv/` virtual environment. All subsequent `make` and `uv run` commands use that environment automatically.

## Configuration

Copy the template and fill in your keys:

```bash
cp .env.example .env
```

| Variable | Required | Default | Description |
|---|---|---|---|
| `GOOGLE_API_KEY` | Yes | — | Google AI Studio API key. Required — the LLM judge calls Gemini. |
| `OPIK_API_KEY` | Yes (evals) | — | Opik API key. **Mandatory for all eval workflows.** Upload, offline, and online evaluation all write to Opik. |
| `OPIK_WORKSPACE` | No | — | Opik workspace name. Only used when `OPIK_API_KEY` is set. |
| `OPIK_PROJECT_NAME` | No | `writing-workflow` | Opik project label. The metric reuses the same project as normal generation traces so all runs colocate. |

Settings are loaded by `src/writing/config/settings.py` via Pydantic Settings. Unknown keys in `.env` are silently ignored.

## Concepts

### Dataset splits

Each entry in `datasets/linkedin_paul_iusztin/index.yaml` carries a `scope:` list that assigns it to one or more named splits:

| Split | Purpose |
|---|---|
| `train_evaluator` | Few-shot examples injected into the judge prompt to calibrate its reasoning. |
| `dev_evaluator` | Alignment development set — used to iterate on the judge prompt. |
| `test_evaluator` | Held-out test set — final validation after dev F1 plateaus. |
| `online_test` | No human labels; emulates production posts that have never been graded. |

### Offline evaluation

The harness reads pre-generated posts stored in the dataset. The judge scores each post (pass / fail). The harness then computes F1 between the judge's binary verdicts and the human-assigned `label` fields. Used to align the judge and measure how well it agrees with human reviewers.

### Online evaluation

The harness generates posts on the fly by running the full writing workflow for each dataset entry, then immediately judges each result. This emulates a production evaluation loop. F1 is computed only when the split has human labels (i.e. NOT `online_test`).

### F1 alignment

F1 is computed between the LLM judge's verdicts and the human-assigned labels in the dataset. A high F1 means the judge reliably agrees with human reviewers. The workflow is: tune the judge prompt and few-shot examples on `dev_evaluator` until F1 plateaus, then validate on `test_evaluator` without further tuning.

## Running the harness

### Make targets

```bash
# Push all labeled splits to Opik (run once before evaluating)
make upload-eval-dataset

# Offline F1 on the dev split (iterate here to align the judge)
make eval-dev

# Offline F1 on the held-out test split (run after dev is stable)
make eval-test

# Online evaluation: generate posts on the fly, judge them
# Prints "Online evaluation complete (no F1 — simulating real-world usage)." for online_test
make eval-online
```

### Direct script invocations

For finer control over split, parallelism, and sample count:

```bash
# Offline evaluation — dev or test split
uv run python scripts/run_evaluation.py \
    --split dev_evaluator \   # or test_evaluator
    --workers 2 \             # parallel evaluation threads
    --nb-samples 5            # limit samples (omit for all)

# Online evaluation
uv run python scripts/run_online_evaluation.py \
    --split online_test \     # or dev_evaluator / test_evaluator
    --workers 1 \             # generation is heavy; 1 is a safe default
    --nb-samples 3            # limit samples (omit for all)
```

See [`scripts/run_evaluation.py`](../../../scripts/run_evaluation.py) and [`scripts/run_online_evaluation.py`](../../../scripts/run_online_evaluation.py) for the full option list.

## Authoring labeled samples

To add a new labeled sample, edit `datasets/linkedin_paul_iusztin/index.yaml` and append an entry:

```yaml
- slug: your-post-slug
  local_post: ./your-post-slug.md
  local_generated_post: ./your-post-slug_generated.md
  local_guideline: ./your-post-slug_guideline.md
  local_research: ./your-post-slug_research.md
  scope:
    - dev_evaluator       # or test_evaluator / train_evaluator / online_test
  label: pass             # or fail — required for offline evaluation and few-shot examples
  critique: "One sentence explaining the verdict."
```

Key rules:

- `label` must be `pass` or `fail`.
- `critique` should explain the verdict in one to two sentences.
- Entries with both `label` and `critique` are eligible as `train_evaluator` few-shot examples **and** flow into any offline evaluation split they are assigned to via `scope`.
- `online_test` entries should not have a `label` field — they emulate unlabeled production data.
- After editing, rerun `make upload-eval-dataset` to push the new entry to Opik, then `make eval-dev` (or `eval-test`) to see how the new sample affects F1.

## Cost considerations

- **Offline evaluation:** each sample triggers at most one Gemini judge call. Cost scales linearly with the number of samples in the split.
- **Online evaluation:** each sample runs the full writing workflow (multiple Gemini calls for drafting + reviewing) plus one Gemini judge call. Online runs are substantially more expensive per sample.
- Use `--nb-samples N` while iterating on the judge prompt to keep cost bounded. Switch to the full split only when validating a stable configuration.

## Troubleshooting

- **"F1 score (judge vs expert labels): 0.000" with a warning about 0 samples** — no labeled entries in the dataset match the requested split filter. Check the `scope:` values in `index.yaml` and confirm they match the `--split` argument exactly (e.g. `dev_evaluator`, not `dev`).
- **`upload_dataset_to_opik` raises `ValueError: No entries found for split '…'`** — either the split name is misspelled or the dataset directory is missing. Verify that `datasets/linkedin_paul_iusztin/` is accessible from `implement_yourself/` (see Prerequisites for the symlink setup).
- **Online runs are slow** — generation is heavy by design. Reduce `--nb-samples` or increase `--workers` (carefully — each worker calls Gemini concurrently). Parallelism beyond 2–3 workers may hit rate limits.
- **`OPIK_API_KEY` not set** — all eval targets will fail. Unlike the rest of the system, Opik is not optional here; every eval target writes results to Opik.
